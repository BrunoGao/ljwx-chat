# 文件上传问题修复说明

## 问题根因

经过深入排查，文件上传失败的根本原因是:

**预签名 URL 使用了内部地址 (`host.docker.internal:9002`), 浏览器无法访问该地址。**

### 技术细节

1. **环境配置**:
   - `S3_ENDPOINT=http://host.docker.internal:9002` - 容器内访问 MinIO 的地址
   - `S3_PUBLIC_DOMAIN=http://192.168.1.83:9002` - 浏览器访问 MinIO 的地址

2. **问题流程**:

   ```
   浏览器 → LobeChat 容器 → 请求预签名 URL
                            → S3 SDK 生成 URL (使用 S3_ENDPOINT)
                            → 返回: http://host.docker.internal:9002/...
   浏览器 → 尝试上传到 host.docker.internal ❌ 失败!
   ```

3. **错误表现**:
   - XMLHttpRequest status = 0 (网络错误)
   - 被识别为 `UPLOAD_NETWORK_ERROR`
   - 显示: "请确认你的网络是否正常，并检查文件存储服务跨域配置是否正确"

## 修复方案

### 代码修改

修改文件: `src/server/modules/S3/index.ts`

在 `createPreSignedUrl` 方法中添加 URL 替换逻辑:

```typescript
public async createPreSignedUrl(key: string): Promise<string> {
  const command = new PutObjectCommand({
    ACL: this.setAcl ? 'public-read' : undefined,
    Bucket: this.bucket,
    Key: key,
  });

  const url = await getSignedUrl(this.client, command, { expiresIn: 3600 });

  // Replace internal endpoint with public domain if configured
  if (fileEnv.S3_PUBLIC_DOMAIN && fileEnv.S3_ENDPOINT) {
    const endpointUrl = new URL(fileEnv.S3_ENDPOINT);
    const publicUrl = new URL(fileEnv.S3_PUBLIC_DOMAIN);
    return url.replace(endpointUrl.origin, publicUrl.origin);
  }

  return url;
}
```

### 修复原理

当同时配置了 `S3_ENDPOINT` 和 `S3_PUBLIC_DOMAIN` 时:

1. SDK 生成的预签名 URL 包含内部地址: `http://host.docker.internal:9002/...`
2. 代码将 `host.docker.internal:9002` 替换为 `192.168.1.83:9002`
3. 浏览器可以访问替换后的 URL: `http://192.168.1.83:9002/...`

## 常见问题

### Q: 重建镜像会丢失数据吗？

A: 不会。数据存储在:

- PostgreSQL 数据库 (host 上的 PostgreSQL)
- MinIO 存储 (./minio-data 目录)

重建只会更新应用代码，不影响数据。

### Q: 构建很慢怎么办？

A: 如果网络较慢，可以使用代理:

```bash
./build-docker-with-proxy.sh
```

### Q: 修复后还是无法上传？

A: 检查:

1. 浏览器控制台 (F12) 的具体错误
2. 预签名 URL 是否包含正确的公网地址
3. 是否可以直接访问 <http://192.168.1.83:9002/lingjingwanxiang/>
4. 防火墙是否阻止了 9002 端口

### Q: 可以直接用 S3_PUBLIC_DOMAIN 作为 S3_ENDPOINT 吗？

A: 不可以。因为:

- 容器内必须通过 `host.docker.internal` 访问 host 上的服务
- `192.168.1.83` 在容器内解析可能不正确
- 这就是需要两个不同配置的原因

## 技术背景

这是 Docker 部署中的常见问题，称为 "内外网地址不一致"。

- **内网地址**: 容器内部通过 `host.docker.internal` 访问 host 服务
- **外网地址**: 浏览器通过局域网 IP 或公网域名访问服务

预签名 URL 是由服务端生成但由浏览器使用的，因此需要使用外网地址。

其他可能遇到类似问题的场景:

- Kubernetes 部署 (内部 Service vs 外部 Ingress)
- 反向代理 (内部 upstream vs 外部 domain)
- 云服务部署 (内网 IP vs 公网 IP)
