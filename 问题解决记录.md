### 第二次报错：403 Forbidden

**时间**: MinIO 启动后
**错误消息**: "Forbidden"
**错误类型**: HTTP 403

#### 诊断过程

1. ✅ MinIO 运行正常 → `curl http://localhost:9002/minio/health/live` 返回 200
2. ✅ Bucket 存在 → `mc ls local/lingjingwanxiang` 显示 bucket
3. ✅ 访问策略正确 → `mc anonymous get` 显示 `public`
4. ✅ CORS 配置正确 → 测试 OPTIONS 请求通过
5. ❌ 预签名 URL 问题 → 深入分析代码

#### 第一次修复尝试（错误）

**方案**: 在生成预签名 URL 后替换域名

```typescript
// ❌ 错误的修复
const url = await getSignedUrl(this.client, command);
if (fileEnv.S3_PUBLIC_DOMAIN && fileEnv.S3_ENDPOINT) {
  return url.replace(
    new URL(fileEnv.S3_ENDPOINT).origin,
    new URL(fileEnv.S3_PUBLIC_DOMAIN).origin
  );
}
```

**问题**: AWS Signature V4 签名基于完整 URL 计算，修改 URL 导致签名失效。

**测试结果**:

- ✅ 预签名 URL 包含正确的公网地址 `192.168.1.83:9002`
- ❌ 上传返回 403 Forbidden（签名失效）

#### 根本原因分析

AWS 签名机制：

```
签名 = HMAC-SHA256(
  Secret Key,
  String to Sign
)

String to Sign 包括:
- HTTP 方法 (PUT)
- 完整 URL (包括 host) ← 关键
- Headers
- Payload hash
```

当生成签名后修改 URL，签名验证失败 → 403 Forbidden

#### 第二次修复（正确）

**方案**: 创建两个独立的 S3Client

```typescript
class S3 {
  private readonly client: S3Client; // S3_ENDPOINT - 内部操作
  private readonly presignClient: S3Client; // S3_PUBLIC_DOMAIN - 预签名 URL

  constructor() {
    this.client = new S3Client({
      ...config,
      endpoint: fileEnv.S3_ENDPOINT, // host.docker.internal:9002
    });

    this.presignClient = new S3Client({
      ...config,
      endpoint: fileEnv.S3_PUBLIC_DOMAIN || fileEnv.S3_ENDPOINT, // 192.168.1.83:9002
    });
  }

  async createPreSignedUrl(key: string) {
    // 使用 presignClient 生成 URL
    return getSignedUrl(this.presignClient, command);
  }
}
```

**优势**:

- ✅ 签名基于正确的 endpoint（公网地址）
- ✅ 内部操作仍使用容器可访问的地址
- ✅ 不需要事后修改 URL

## 🔍 技术深度分析

### 为什么需要两个不同的地址？

#### Docker 网络架构

```
┌─────────────────────────────────────┐
│         Host Machine                 │
│                                      │
│  ┌────────────────┐  ┌────────────┐ │
│  │ MinIO Container│  │ App Container││
│  │ Port: 9000     │  │ Port: 3210   ││
│  │ → :9002        │  │              ││
│  └────────────────┘  └────────────┘ │
│         ↑                  ↑         │
│         │                  │         │
└─────────┼──────────────────┼─────────┘
          │                  │
    (host.docker.internal:9002)
          │                  │
┌─────────┴──────────────────┴─────────┐
│         Browser                       │
│    192.168.1.83:3210                 │
│    192.168.1.83:9002                 │
└───────────────────────────────────────┘
```

**关键点**:

- 容器内：通过 `host.docker.internal` 访问 host 服务
- 浏览器：通过 `192.168.1.83` 访问 host 服务
- 预签名 URL: 由服务端生成，但由浏览器使用

### AWS S3 预签名 URL 工作流程

1. **客户端请求**:

   ```
   Browser → LobeChat: "我要上传文件"
   ```

2. **服务端生成预签名 URL**:

   ```typescript
   const command = new PutObjectCommand({...});
   const url = await getSignedUrl(presignClient, command);
   // url = http://192.168.1.83:9002/bucket/file?X-Amz-Signature=...
   ```

3. **签名计算**:

   ```
   Canonical Request =
     "PUT\n" +
     "/bucket/file\n" +
     "X-Amz-Algorithm=AWS4-HMAC-SHA256&...\n" +
     "host:192.168.1.83:9002\n" +    ← 包含在签名中
     ...

   Signature = HMAC-SHA256(secretKey, stringToSign)
   ```

4. **客户端使用**:
   ```
   Browser → PUT http://192.168.1.83:9002/bucket/file?X-Amz-Signature=...
   MinIO → 验证签名 → ✅ 通过
   ```

### 为什么简单替换 URL 会失败？

```typescript
// 错误方案
const url = await getSignedUrl(client, command);
// url = http://host.docker.internal:9002/...?X-Amz-Signature=abc123
// 签名基于: host:host.docker.internal:9002

const fixedUrl = url.replace('host.docker.internal', '192.168.1.83');
// fixedUrl = http://192.168.1.83:9002/...?X-Amz-Signature=abc123
// 但签名仍然是基于 host.docker.internal 计算的！

Browser → PUT http://192.168.1.83:9002/...?X-Amz-Signature=abc123
MinIO → 验证签名:
  期望的 host: 192.168.1.83:9002
  签名中的 host: host.docker.internal:9002
  → ❌ 签名不匹配 → 403 Forbidden
```

### 双客户端方案的优雅之处

```typescript
// 正确方案
const presignClient = new S3Client({
  endpoint: 'http://192.168.1.83:9002', // 直接使用公网地址
});

const url = await getSignedUrl(presignClient, command);
// url = http://192.168.1.83:9002/...?X-Amz-Signature=xyz789
// 签名基于: host:192.168.1.83:9002 ← 正确！

Browser → PUT http://192.168.1.83:9002/...?X-Amz-Signature=xyz789
MinIO → 验证签名:
  期望的 host: 192.168.1.83:9002
  签名中的 host: 192.168.1.83:9002
  → ✅ 匹配 → 200 OK
```

## 🎓 经验教训

### 1. Docker 网络的复杂性

在 Docker 环境中，需要区分：

- 容器间通信（内部网络）
- 容器与 host 通信（host.docker.internal）
- 外部客户端访问（公网 IP / 域名）

### 2. 云服务 API 的签名机制

修改已签名的 URL 通常会导致签名失效，应该：

- 在生成签名时使用正确的 endpoint
- 而不是事后修改 URL

### 3. 分层诊断的重要性

问题排查应该层层递进：

1. 服务是否运行？
2. 网络是否连通？
3. 权限是否正确？
4. 签名是否有效？

### 4. 测试的重要性

每次修复后都应该：

- 运行自动化测试
- 进行端到端测试
- 验证边界情况
