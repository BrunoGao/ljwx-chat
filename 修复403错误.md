# 修复 403 Forbidden 错误

## 问题分析

之前的修复方案有误，导致新的问题：

### 错误的方案（已修复）

```typescript
// ❌ 错误：简单替换 URL 会破坏 AWS 签名
const url = await getSignedUrl(this.client, command);
return url.replace('host.docker.internal:9002', '192.168.1.83:9002');
```

**问题**: AWS 签名是基于完整 URL 计算的，替换域名后签名失效，导致 403 Forbidden。

### 正确的方案（已实施）

创建两个 S3Client：

- `client`: 使用 `S3_ENDPOINT` (内部地址) - 用于实际的 S3 操作
- `presignClient`: 使用 `S3_PUBLIC_DOMAIN` (公网地址) - 用于生成预签名 URL

```typescript
// ✅ 正确：使用不同的 client 生成预签名 URL
export class S3 {
  private readonly client: S3Client; // 用于内部操作
  private readonly presignClient: S3Client; // 用于生成预签名 URL

  constructor() {
    // client 使用 S3_ENDPOINT (host.docker.internal:9002)
    this.client = new S3Client({
      ...config,
      endpoint: fileEnv.S3_ENDPOINT,
    });

    // presignClient 使用 S3_PUBLIC_DOMAIN (192.168.1.83:9002)
    this.presignClient = new S3Client({
      ...config,
      endpoint: fileEnv.S3_PUBLIC_DOMAIN || fileEnv.S3_ENDPOINT,
    });
  }

  async createPreSignedUrl(key: string): Promise<string> {
    // 使用 presignClient 生成 URL
    return getSignedUrl(this.presignClient, command, { expiresIn: 3600 });
  }
}
```

## 技术细节

### 为什么需要两个 Client？

AWS SDK 在生成预签名 URL 时，会使用创建 S3Client 时指定的 endpoint 来构建 URL，并基于这个 URL 计算签名。

- 如果使用内部地址 (`host.docker.internal:9002`)，浏览器无法访问
- 如果全部使用公网地址 (`192.168.1.83:9002`)，容器内部操作会失败

因此需要两个客户端，各司其职。

### AWS 签名机制

AWS Signature V4 签名包括：

1. HTTP 方法 (PUT, GET, etc.)
2. 完整的 URL (包括 host)
3. Headers
4. Payload

任何修改都会导致签名失效。
